# E0477 in `async fn` trait impls тАФ investigation log

This crate is a preserved exploration artifact. It is **not** a workspace
member and does not build in CI. It documents the empirical path that led to
removing `OwnedView`'s manual `unsafe impl<V: Send + 'static> Send` in favour
of auto-trait derivation.

The investigation started from a suggestion that `#[trait_variant::make(Send)]`
might let service implementors write `async fn` instead of the
`fn -> impl Future + Send { async move { ... } }` workaround. It ended
somewhere quite different.

## The problem

Service traits generated by connectrpc-codegen have this shape:

```rust
pub trait ElizaService: Send + Sync + 'static {
    fn say(&self, ctx: Context, req: OwnedView<SayRequestView<'static>>)
        -> impl Future<Output = Result<(SayResponse, Context), ConnectError>> + Send;
}
```

Implementing with `async fn` hits [rust-lang/rust#128095]:

```text
error[E0477]: the type `SayRequestView<'_>` does not fulfill the required lifetime
note: type must satisfy the static lifetime as required by this binding
   |     ... -> impl Future<Output = ...> + Send;
   |                                        ^^^^
```

Users had to wrap in `fn -> impl Future + Send { async move { ... } }`.

## What was tried, in order

All results on rustc 1.77.2 stable and 1.94.0-nightly (2025-12-05).

### trait_variant тАФ no help

| Feature | What | Result |
|---|---|---|
| `repro_a` | `async fn` vs our `fn -> impl Future + Send` trait | E0477 |
| `repro_b` | `async fn` vs `#[trait_variant::make(Send)]` trait | **E0477, identical** |
| `repro_c` | Dual-trait form, impl the non-`Send` local variant | Compiles but doesn't satisfy the `Send` variant тАФ blanket impl goes the wrong direction |

`cargo expand` on case B showed `trait_variant::make(Send)` expands to
**exactly** the signature we already generate. It is a trait-definition
transform; nothing changes at the impl site.

### return_type_notation тАФ no help (and a near miss)

RTN cases were removed from this crate because the `(..)` syntax is
parse-gated and can't be `cfg`-stripped. Findings:

- **RTN on trait definition** (`where Self::handle(..): Send`): same E0477.
  The bound is still an impl obligation.
- **RTN on consumer** (`where S::handle(..): Send` on the dispatch fn): trades
  E0477 for *"implementation of `Send` is not general enough"* тАФ `(..)`
  quantifies over all input lifetimes including `&'_ self`, and the async body
  only proves `Send` for the specific lifetime it's called with.
- **False positive warning**: an early RTN-on-consumer test *appeared* to pass
  because the call site used `todo!()` as an argument. The `!` coercion made
  the call unreachable and the obligation was skipped. Replacing `todo!()`
  with a real construction path surfaced the HRTB error. **Don't use `todo!()`
  in type-level probes.**

### Body-side workarounds тАФ none exist

| Feature | Body | Result |
|---|---|---|
| `drop_g1` | `drop(req)` after using across await | E0477 |
| `drop_g2` | `let req = req;` rebind | E0477 |
| `drop_g3` | `drop(req)` before any await | E0477 |
| `drop_g4` | Baseline (no workaround) | E0477 |
| `drop_g5` | Consume+shadow line 1 (mimics `to_owned_message()`) | E0477 |
| `drop_g6` | **Empty body**, `_req` unused | **E0477** |
| `drop_g7` | `_` pattern тАФ argument never bound | E0477 |

`drop_g6` is the decisive case: the error fires at **signature desugaring**
before the body is examined. This also invalidated our prior documentation,
which claimed consuming the `OwnedView` before any `.await` let you use
`async fn`. It never did.

The #128095 thread mentions `drop()` as a workaround тАФ it presumably helps
repros where the issue is generator-upvar analysis, not signature desugaring.
That is a different flavor of the bug.

### `Cow<'static, str>` тАФ the clue

| Feature | Parameter type | Result |
|---|---|---|
| `cow_h` | `Cow<'static, str>` | **OK** |

`Cow` is auto-`Send`. It has no manual impl with a `'static` bound. This
narrowed the trigger from "any type with a `'static` lifetime parameter" to
"the `V: 'static` bound on `OwnedView`'s conditional `Send` impl specifically."

### The fix тАФ drop the manual impl

`OwnedView<V>` is `ManuallyDrop<V>` + `Bytes`. Both forward auto-traits.
`OwnedView<V>: Send iff V: Send` follows тАФ no `'static` bound needed.

The manual impl's `V: 'static` was defensive: it blocked
`OwnedView<FooView<'short>>: Send` where the view might borrow from outside
`self.bytes`. But that type is **unconstructible**: all constructors are
gated on `V: MessageView<'static>` and fields are private. The bound guarded
against a value that can't exist.

| Feature | What | Result |
|---|---|---|
| `repro_a,auto_send` | async fn, borrow across await | **OK** |
| `drop_g1,auto_send` | drop after await | **OK** |
| `drop_g6,auto_send` | empty body | **OK** |
| `cross_old` | six cross-boundary patterns, manual impl | **5/6 E0477** |
| `auto_send` | same six patterns, auto-trait | **6/6 OK** |

`cross.rs` verified that the fix composes: `&req` across await, helper fns
by value and by ref, generic helpers, nested trait impls тАФ all work. The old
manual `Sync` impl had the same `'static` bound, so `&req` across await was
*also* broken before (silently тАФ nothing documented this).

## Re-running

```sh
# Baseline (workaround compiles)
cargo check

# Reproduce the failure
cargo check --features repro_a        # E0477

# Verify the fix
cargo check --features repro_a,auto_send   # OK

# Sweep all drop variants (before / after)
for f in drop_g1 drop_g2 drop_g3 drop_g4 drop_g5 drop_g6 drop_g7; do
  printf '%-8s  manual:' "$f"
  cargo check --features "$f" 2>&1 | grep -q '^error' && printf ' E0477 ' || printf ' OK    '
  printf ' auto:'
  cargo check --features "$f,auto_send" 2>&1 | grep -q '^error' && echo ' E0477' || echo ' OK'
done
```
